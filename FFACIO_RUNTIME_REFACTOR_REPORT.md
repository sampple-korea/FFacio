# FFacio Runtime Demo 정렬 수정 보고서

## 적용 버전

- 앱 버전: `0.6.2-runtime-demo-aligned-final`
- 얼굴 처리 엔진: FFacio Runtime Binder 경로만 사용
- 등록 템플릿: 사용자당 대표 Runtime 템플릿 1개
- 인증 안정화: 1프레임

## 이번 수정의 핵심

기존 FFacio에 별도로 들어 있던 능동 고개 돌리기 챌린지, 정면·좌·우 5장 다각도 등록, 자세별 유지 시간, 반복 자세 판정, 등록 샘플 간 유사도 재검사, 보조 샘플 지지 개수 판정을 모두 제거했다. 화면 문구만 숨긴 것이 아니라 상태 변수, 판정 함수, 저장·로딩, 인증 비교, 테스트에서 해당 경로를 없앴다.

등록은 Runtime Demo의 등록 검사와 같은 기본 조건을 통과한 가장 큰 얼굴을 1.2초 동안 안정적으로 관찰한 뒤, 그 구간에서 품질이 가장 좋은 Runtime 템플릿 하나만 저장한다. 여러 얼굴이 화면에 있어도 얼굴 사각형 면적이 가장 큰 얼굴 하나만 등록·인증 대상으로 삼는다.

인증은 Runtime 라이브니스(켜진 경우), 얼굴 품질, 최소 얼굴 면적, Runtime 유사도 0.80, 1위와 2위의 점수 차이 0.03을 사용한다. 별도의 고개 동작이나 다중 샘플 지지 판정은 하지 않으며, 조건을 통과한 첫 프레임에서 인증한다.

## Runtime Demo 기본값 정렬

- 라이브니스 사용: 켜짐
- 라이브니스 레벨: 0
- 라이브니스 임계값: 0.70
- 얼굴 식별 임계값: 0.80
- 1위·2위 불확실 구간: 0.03
- 품질 임계값: 0.50
- 밝기 범위: 0~255
- 최대 yaw/pitch/roll: 각각 10도
- 왼쪽·오른쪽 눈 감김 임계값: 각각 0.80
- 얼굴 가림 검사: 기본 꺼짐
- 얼굴 가림 임계값: 0.50
- 입 벌림 임계값: 0.50
- 등록 얼굴 크기: 80~1200px
- 등록 안정화 시간: 1200ms
- 분석 프레임 간격: 180ms
- 인증 결과 유지: 3500ms
- 인증 안정화 프레임: 사용자 요구에 따라 Demo 기본 3 대신 1

관리자 고급 설정에서는 Runtime 라이브니스 사용 여부, Runtime 라이브니스 레벨, 얼굴 가림 검사 사용 여부를 바꿀 수 있다. 라이브니스를 끈 경우 예전 고개 돌리기 방식으로 대체하지 않고 품질·유사도 기준만 사용한다.

## 기존 등록 데이터 처리

사용자 요청에 따라 이전 등록 데이터는 호환하거나 변환하지 않는다. 이 버전을 처음 실행하면 과거 사용자 레코드와 암호화된 사용자 템플릿을 모두 삭제하고 저장 정책 버전 5를 기록한다. 릴레이 URL·토큰은 유지하되 얼굴 인식 설정은 Runtime Demo 기본값(라이브니스 켬·레벨 0·가림 검사 끔)으로 초기화된다. 이후에는 스키마 5의 단일 `template_b64` 레코드만 저장·로딩하므로 과거 다각도 샘플이나 레거시 대표 템플릿이 인증에 섞일 수 없다.

## 유지한 안전 장치

- Runtime 템플릿 형식·크기 검사
- 기존 사용자와의 중복 얼굴 비교 및 경고(등록 자체는 Demo처럼 허용)
- 등록 시 중앙 위치, 크기, 밝기, 정면 각도, 68점 특징점, 눈, 선택적 가림, 입 상태 검사
- 인증 시 Runtime 비교 오류와 일반 미인식 구분
- 암호화 로컬 저장소, Head Admin, 카메라 watchdog, Runtime 재연결, 릴레이 보안 정책

기존 사용자 데이터는 첫 실행 초기화로 전부 제거되므로 모든 사용자를 새 방식으로 다시 등록해야 한다.

Runtime Demo 클라이언트와의 성능 비교에서, 실제 앱에만 있던 프레임별 임시 파일 덮어쓰기와 동기식 `fsync`를 제거했다. 임시 YUV/PNG는 앱 전용 캐시에서 Demo처럼 즉시 삭제한다. 저장 중 화면 종료와 템플릿 정리가 충돌하지 않도록 저장 전 독립 템플릿 스냅샷을 만들며, 손상된 개별 사용자 레코드는 다른 정상 사용자의 로딩을 막지 않고 제외한다.

## 검증 항목

`python3 scripts/verify_source_static.py`는 다음을 fail-closed 방식으로 검사한다.

- Kotlin/Java/Gradle 괄호·문자열 구조와 XML 파싱
- 자체 ONNX/OpenCV 엔진 및 모델 파일 부재
- Runtime 검출·템플릿 추출·유사도 호출 존재
- 가장 큰 얼굴 선택, 단일 등록 안정화, 1프레임 인증 상수 존재
- 폐기한 다각도·고개 챌린지·보조 샘플·샘플 지지 로직 표식 부재
- 비밀키·서명키 파일 부재

단위 테스트도 가장 큰 얼굴 선택, 단일 템플릿 안정화, 잘못된 프레임에서 안정화 초기화, 인증 1프레임, 대표 템플릿 단독 비교 기준으로 교체했다. 최종 순수 정책 선언과 저장소 테스트 79개를 Kotlin/JVM 17로 함께 컴파일해 79개 모두 통과하는 것까지 확인했다.

## 빌드 조건

FFacio와 FFacio Runtime은 signature 권한을 사용하므로 실제 배포 APK는 같은 인증서로 서명해야 한다. 완전한 Android SDK와 Gradle/Maven 의존성이 있는 환경에서는 `android/gradlew testDebugUnitTest lintDebug assembleDebug`로 최종 빌드·테스트할 수 있다.
